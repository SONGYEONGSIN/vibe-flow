# TDD (Test-Driven Development)

## Iron Law

**테스트를 먼저 쓰고, 실패를 확인한 뒤에만 구현 코드를 작성한다.**

이것은 권장이 아니라 규칙이다. 위반 시 테스트 없이 작성된 코드는 삭제 대상이다.

## RED-GREEN-REFACTOR

모든 기능 구현은 이 3단계 사이클을 따른다:

### 1. RED — 실패하는 테스트 작성
- 구현하려는 기능의 테스트를 먼저 작성한다
- 테스트를 실행하여 **실패하는 것을 확인**한다 (이것이 RED)
- 항상 통과하는 테스트는 아무것도 검증하지 않으므로 무의미하다

### 2. GREEN — 최소 구현
- 테스트를 통과시키기 위한 **최소한의 코드**만 작성한다
- 완벽한 코드가 아니어도 된다. 테스트가 통과하면 충분하다
- 테스트를 다시 실행하여 **통과하는 것을 확인**한다

### 3. REFACTOR — 정리
- 테스트가 통과하는 상태에서 코드를 정리한다
- 중복 제거, 네이밍 개선, 구조 개선
- 리팩토링 후 테스트가 여전히 통과하는지 확인한다

## 예외

다음 경우에만 RED-GREEN-REFACTOR 사이클을 적용하지 않는다:
- 설정 파일 변경 (`.json`, `.yaml`, `.env` 등)
- 스타일 전용 변경 (CSS/Tailwind 클래스만 수정)
- 문서 변경 (`.md` 파일)
- 타입 정의만 변경 (`types.ts` 등 런타임 영향 없음)

**단, TDD 예외라도 검증 자체는 필수다.** 설정 변경 시 빌드 확인, 스타일 변경 시 렌더링 확인 등 변경 영향을 반드시 확인한다.

### Production Incident 예외 (좁은 범위)

**조건 (모두 충족해야 적용)**:
- 프로덕션 환경에서 실제 고객 영향이 발생 중 (서비스 다운, 데이터 손상, 결제 실패 등)
- 인시던트 채널/티켓에 영향 범위 + 시작 시간이 명시됨
- 수정자가 "5분 안에 배포해야 하는가" 자문하여 그렇다고 답할 수 있음

**적용 시 절차**:
1. **수정 → 배포** 우선 (테스트 작성 생략 허용)
2. 배포 즉시 인시던트 채널에 "테스트 미작성 핫픽스" 명시
3. **24시간 내 회귀 테스트 작성 + PR 머지 필수** (안 하면 다음 인시던트 시 예외 권한 박탈)
4. 인시던트 회고에 "왜 RED 단계가 사전에 없었는가" 분석 포함

**적용 안 됨**:
- "급한 기능 출시" — 출시 일정 압박은 인시던트 아님
- "내일까지 끝내야 하는 PoC" — 일정 문제는 TDD 지킬 수 있음
- "오래된 코드라 테스트하기 어려움" — 어렵다고 예외 아님

## 위반 패턴

- ❌ 구현 코드를 먼저 쓰고 나중에 테스트 추가 (test-last)
- ❌ 테스트 없이 "동작 확인했습니다" 선언
- ❌ RED 단계를 건너뛰고 바로 GREEN (테스트가 처음부터 통과)
- ❌ REFACTOR 단계에서 테스트 재실행 생략
