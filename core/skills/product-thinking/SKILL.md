호소다 다카히로 *컨셉수업*의 5 프레임워크 reference. **컨셉이 막히거나 흐릿할 때** 적용해 좋은 질문 → 좋은 컨셉으로 이행한다.

## 사용 시점

**필수**:
- 제품/기능 기획 **초기** — 컨셉 잡기 단계
- 기존 컨셉이 흐릿하거나 임팩트 약하다고 느껴질 때
- 경쟁 분석이 같은 범주에만 머물러 시야 좁다고 느껴질 때
- 가치 제안이 팩트에 머무르고 베네핏까지 못 갈 때

**스킵 가능**:
- 이미 컨셉/사용자/문제 명확 → `/brainstorm`이나 `/plan`으로 직행
- 단순 버그 수정, 리팩토링
- 큰 컨셉 결정 없는 작은 기능 추가

## 적용 프레임워크 (선택)

상황에 따라 1~5개 골라 적용. 모두 동시에 적용할 필요 없음. 자세한 본문은 `references/concept-lesson.md`.

| # | 프레임워크 | 한 줄 | 사용 상황 |
|---|----------|-------|---------|
| 1 | 질문 설계 공식 | 좋은 질문 = 자유도 × 임팩트 | 핵심 질문 평가/재구성 |
| 2 | 재구성 7가지 | 부분→전체 / 객관→주관 / 현실→이상 / 명사→동사 / 만들기→부수기 / 수단→목적 / 이기적→이타적 | 컨셉 막혔을 때 관점 전환 |
| 3 | 4C 스토리 | Customer → Competitor → Company → Concept | 피치덱·기획서 설득력 검증 |
| 4 | 경쟁자 3층 분석 | 같은 범주 / 같은 과제(JTBD) / 같은 시간 | 블루오션·기회 탐색 |
| 5 | 팩트 → 메리트 → 베네핏 | 객관 사실 → 일반 이익 → 타깃 특화 이익 | 가치 제안 작성 |

## 절차

### 1. 컨셉/질문 식별
사용자가 작업 중인 제품/기능에서 핵심 질문을 한 문장으로 추출. 예: "어떻게 하면 사용자가 X를 더 잘하게 할까?"

### 2. 프레임워크 1~2개 선택
상황과 막힌 지점에 맞춰 위 5개 중 1~2개 선택. 5개 다 적용은 과잉.

### 3. 적용 → 후보 도출
선택된 프레임워크를 단계별로 적용. 각 프레임워크가 만든 새 후보(질문/컨셉/스토리/경쟁/베네핏) 1~3개씩 나열.

### 4. 결정
원안 + 신규 후보를 비교하여 어떤 것을 채택할지 결정. **자유도 × 임팩트 2축**으로 평가하면 객관적.

### 5. 다음 스킬로 이행
컨셉이 명확해지면:
- `/brainstorm`으로 구현 의도/제약 탐색 진입
- `/plan`으로 멀티스텝 작업 분해
- `/idea`로 사업 아이디어 검증 (망할 가능성 먼저)

```markdown
<적용 결과 + 새 후보들>

| 후보 | 자유도 | 임팩트 |
|------|--------|--------|

| 후보A | 고 | 중 |

### 채택
<채택 후보 + 사유>

<제안 스킬>
```
