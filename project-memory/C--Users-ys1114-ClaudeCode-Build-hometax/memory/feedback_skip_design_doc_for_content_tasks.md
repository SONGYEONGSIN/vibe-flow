코드 프로젝트가 아닌 컨텐츠·자문·분석 보고 작업(예: 홈택스 자문단 회신, 의견서 작성)에선 design.md 스펙 파일과 writing-plans 단계를 만들지 말고, 브레인스토밍에서 합의된 프레임을 대화 컨텍스트에서만 활용해 바로 실행한다.

**Why:** 산출물이 텍스트(양식·의견서)일 때는 메타 문서(스펙→플랜→실행)가 본 작업보다 무거워져 YAGNI 위반이 됨. 사용자가 "design.md 파일이 필요한가?" 라고 직접 지적함 (2026-05-06).

**How to apply:**
- 트리거: 사용자가 "양식 작성", "의견서", "자문 회신", "보고서 본문", "글 다듬기" 등 코드가 아닌 텍스트 산출물을 요청할 때
- 브레인스토밍은 대화로만 진행, 합의된 프레임/체크리스트는 작업 노트(raw-notes.md 등)에 가볍게 메모
- 단, 코드/시스템 구현 작업은 기존대로 design.md → writing-plans 흐름 유지
- 헷갈리면 "이건 컨텐츠 작업이라 스펙 문서는 생략할게, 바로 들어가도 될까?" 한 번 확인
