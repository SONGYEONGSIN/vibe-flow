# 프로젝트 회고 워크플로우

세션 노트(`.claude/session-notes.md`)와 git 이력을 기반으로 프로젝트 회고를 생성한다.
**세션 컨텍스트가 없어도 동작한다** — 파일 기반이므로 세션 유실과 무관.

---

## Phase 1: 소스 수집

아래를 **병렬로** 실행한다.

### 1-1. 세션 노트 읽기

```
Read: .claude/session-notes.md
```

- 파일이 없거나 비어있으면 "세션 노트 없음"으로 표시하고 계속 진행
- `/commit` Phase 4에서 자동 생성된 파일

### 1-2. Git 이력 수집

```bash
# 프로젝트명 추출
basename $(pwd)

# 전체 커밋 이력 (기간 제한 없음 — 마지막 /retro 이후 전부)
git log --oneline --stat

# 주요 변경 파일
git log --name-only --pretty=format:"" | sort | uniq -c | sort -rn | head -20
```

### 1-3. 기존 회고 확인

```
Glob: ~/knowledge/retrospectives/*-{프로젝트명}*.md
```

- 같은 프로젝트의 이전 회고가 있으면 참고 (반복 등장하는 지식 = 공통 지식 후보)

### 1-4. 기술 스택 파악

```
Read: package.json (dependencies, devDependencies만)
```

## Phase 2: 지식 분류

수집한 소스에서 지식을 **2가지 버킷**으로 분류한다.

### 공통 지식 (cross-project)

다른 프로젝트에서도 재사용 가능한 것:

| 유형 | 예시 |
|------|------|
| 프레임워크 패턴 | "Next.js App Router에서 외부 이미지는 remotePatterns 필수" |
| 라이브러리 동작 | "Stripe 웹훅은 raw body 필요" |
| 디버깅 기법 | "Supabase RLS 디버깅: service_role로 우회 테스트" |
| 아키텍처 패턴 | "낙관적 업데이트 + 에러 시 롤백" |
| 도구 사용법 | "Vercel Edge Function 콜드스타트 ~50ms" |

### 프로젝트 고유 지식 (project-specific)

이 프로젝트에서만 의미 있는 것:

| 유형 | 예시 |
|------|------|
| 도메인 로직 | "리워드 금액은 VAT 포함 표시" |
| 비즈니스 규칙 | "펀딩 마감 후 환불 불가" |
| 데이터 모델 | "users와 rewards는 N:M 관계" |
| 외부 연동 | "와디즈 API 인증은 OAuth2 + 별도 파트너 키" |

### 분류 기준

**"다른 프로젝트에서 같은 상황을 만나면 도움이 되는가?"**
- Yes → 공통 지식
- No → 프로젝트 고유

## Phase 3: 회고 문서 생성

### 저장 경로

```
~/knowledge/retrospectives/YYYY-MM-DD-{프로젝트명}.md
```

- `~/knowledge/retrospectives/` 디렉토리가 없으면 생성
- 같은 날 같은 프로젝트의 회고가 이미 있으면 `-2`, `-3` 접미사 추가

### 문서 구조

```markdown
# {프로젝트명} 회고 (YYYY-MM-DD)

## 프로젝트 개요
- **기술 스택**: Next.js, Supabase, Stripe ...
- **작업 기간**: 첫 커밋 ~ 마지막 커밋 날짜
- **커밋 수**: N개

## 새로 알게 된 것
(세션 노트 + git 이력에서 추출한 학습 내용)
- 항목별 1~2줄
- why가 있으면 괄호로 부연

## 공통 지식 (다음 프로젝트에도 쓸 것)
(공통 지식 버킷에서 선별)
- ~/.claude/rules/ 승격 후보에는 [rules 후보] 태그

## 프로젝트 고유 지식
(프로젝트 고유 버킷)
- 도메인, 비즈니스 규칙, 데이터 모델 등

## 삽질 & 교훈
(시간 많이 소요된 디버깅, 시행착오)
- 문제 → 원인 → 해결 형태로 기록

## 개선할 점
(다음 프로젝트에서 프로세스적으로 바꿀 것)
- 워크플로우, 도구, 접근 방식 관련
```

### 작성 규칙

- 한 줄에 하나의 지식. 장황하게 쓰지 않는다
- 코드 블록은 꼭 필요한 경우만 (1~2줄 인라인 선호)
- 추측 금지. 세션 노트와 git에서 확인된 사실만
- 빈 섹션은 삭제 (내용 없는 섹션 남기지 않음)

---

## Phase 4: 세션 노트 정리

회고 생성이 완료되면:

1. `.claude/session-notes.md` 내용을 **비운다** (파일 자체는 유지)
2. 비우기 전 내용은 회고 문서에 이미 반영되었으므로 유실 없음

---

## Phase 5: 보고

```
## 회고 완료

### 저장 위치
~/knowledge/retrospectives/YYYY-MM-DD-{프로젝트명}.md

### 요약
- 새로 알게 된 것: N건
- 공통 지식: N건 (rules 승격 후보 N건)
- 프로젝트 고유: N건
- 삽질 & 교훈: N건

### rules 승격 후보
(있으면 목록 표시 — 사용자가 승격 여부 결정)
- "항목 내용" → ~/.claude/rules/에 추가 제안
```

---

## 금지 사항

- 세션 노트를 회고 **전에** 삭제 금지
- git에 없는 내용을 **추측으로 생성** 금지
- 회고 문서를 **자동 커밋** 금지
- `~/.claude/rules/`를 **자동 수정** 금지 (승격은 사용자 승인 후)
- 이전 회고를 **덮어쓰기** 금지 (항상 새 파일 생성)

## Troubleshooting

### 세션 노트가 없음
- **원인**: `/commit` 시 캡처할 만한 지식이 없었거나, 수동 커밋 사용
- **해결**: git log만으로 회고 생성. "세션 노트 없이 git 이력만으로 생성" 명시

### 커밋이 없음
- **원인**: 프로젝트 초기 또는 다른 브랜치에서 작업
- **해결**: `git log --all`로 전체 브랜치 확인. 그래도 없으면 "회고할 이력이 없습니다" 출력 후 종료

### 회고가 너무 짧음
- **원인**: 작은 프로젝트, 짧은 작업 기간
- **해결**: 정상. 짧아도 기록의 가치가 있다. 억지로 늘리지 않는다

### 같은 프로젝트 반복 회고
- **원인**: 장기 프로젝트에서 여러 번 /retro 실행
- **해결**: 이전 회고 참조하여 중복 내용 제외. 새로운 지식만 기록
