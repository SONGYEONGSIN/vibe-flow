# Consensus Patterns

에이전트 토론에서 합의를 도출하는 패턴, 교착 상태 해소, 결과 품질 보증 기법.

---

## 합의 모델 분류

### 모델 비교

| 모델 | 설명 | 장점 | 단점 | 사용 시기 |
|------|------|------|------|----------|
| 만장일치 | 모든 에이전트 동의 | 최고 품질 합의 | 시간 오래 걸림 | 핵심 아키텍처 결정 |
| 초다수결 | 2/3 이상 동의 | 높은 정당성 | 소수 의견 무시 가능 | 기술 스택 선정 |
| 다수결 | 과반 동의 | 빠른 결정 | 분열 가능 | 빠른 결정 필요 시 |
| 합의 수렴 | 점진적 접근 | 모든 관점 반영 | 타협적 결과 | 이해관계자 多 |
| 권위 결정 | 모더레이터 최종 판단 | 교착 해소 | 토론 의미 약화 | 데드라인 임박 |

## Pattern 1: 점진적 수렴 (Gradual Convergence)

각 라운드마다 합의 영역을 넓혀가는 방식.

### 프로세스

```
Round 1: 전체 범위 탐색
  에이전트 A: [관점 1, 2, 3, 4, 5]
  에이전트 B: [관점 2, 3, 6, 7, 8]
  에이전트 C: [관점 1, 3, 5, 9, 10]
  → 교집합 발견: {3} (공통), {1, 2, 5} (2명 이상)

Round 2: 교집합 확장
  합의 확정: {3}
  토론 대상: {1, 2, 5, 6, 7, 8, 9, 10}
  각 에이전트: 상대 관점의 타당성 평가 (1-5점)
  → 4점 이상 관점 합의에 추가

Round 3: 최종 정리
  합의: {1, 2, 3, 5, 7}
  미합의: {4, 6, 8, 9, 10}
  미합의 항목: "조건부 합의" 또는 "추후 검토"로 분류
```

### 구현 예시

```typescript
interface Position {
  agent: string;
  claim: string;
  confidence: number;  // 0-1
  evidence: string[];
}

interface ConsensusResult {
  agreed: string[];
  conditional: { claim: string; condition: string }[];
  disagreed: { claim: string; positions: Position[] }[];
  confidence: number;
}

function calculateConsensus(
  rounds: Position[][],
  threshold: number = 0.7,
): ConsensusResult {
  const allClaims = new Map<string, Position[]>();

  // 모든 주장 수집
  for (const round of rounds) {
    for (const pos of round) {
      const existing = allClaims.get(pos.claim) ?? [];
      existing.push(pos);
      allClaims.set(pos.claim, existing);
    }
  }

  const agreed: string[] = [];
  const conditional: { claim: string; condition: string }[] = [];
  const disagreed: { claim: string; positions: Position[] }[] = [];

  for (const [claim, positions] of allClaims) {
    const avgConfidence = positions.reduce((sum, p) => sum + p.confidence, 0) / positions.length;
    const supportRatio = positions.filter(p => p.confidence > 0.5).length / positions.length;

    if (supportRatio >= threshold && avgConfidence > 0.7) {
      agreed.push(claim);
    } else if (supportRatio >= 0.5) {
      conditional.push({
        claim,
        condition: `${Math.round(supportRatio * 100)}% 지지, 추가 검토 필요`,
      });
    } else {
      disagreed.push({ claim, positions });
    }
  }

  return {
    agreed,
    conditional,
    disagreed,
    confidence: agreed.length / allClaims.size,
  };
}
```

## Pattern 2: 델파이 기법 (Delphi Method)

익명 의견 수렴 → 피드백 → 재수렴. 권위 편향 제거.

### 프로세스

```
Phase 1: 독립 응답
  각 에이전트가 독립적으로 분석 (다른 에이전트 결과 미참조)
  산출물: 개별 추천안 + 근거 + 확신도(1-10)

Phase 2: 익명 공유
  모든 응답을 익명화하여 전체 공유
  각 에이전트: 다른 응답 검토 후 자기 입장 수정
  변경 시 변경 이유 기록

Phase 3: 수렴 확인
  수렴 기준: 표준편차 < 1.5 (10점 척도)
  미수렴 시: Phase 2 반복 (최대 3회)

Phase 4: 최종 종합
  중앙값 기반 최종 결론
  이상치(outlier) 의견은 "소수 의견"으로 기록
```

### 수렴 측정

```
Round 1 점수: [8, 3, 7, 6, 9]
  평균: 6.6, 표준편차: 2.2 → 미수렴

Round 2 점수: [7, 5, 7, 6, 8]
  평균: 6.6, 표준편차: 1.1 → 수렴!

수렴 속도 지표:
  σ(Round N) / σ(Round 1) < 0.5 → 양호한 수렴
  3라운드 후 σ > 2.0 → 근본적 의견 차이 → 교착 처리
```

## Pattern 3: RAPID 프레임워크

역할 기반 의사결정으로 책임 명확화.

```
R (Recommend): 추천자
  - 데이터 수집, 분석, 옵션 제시
  - 1-3개 추천안 + 각각의 장단점

A (Agree): 동의자
  - 추천안에 대한 거부권
  - 거부 시 구체적 대안 제시 의무

P (Perform): 실행자
  - 결정 사항 실행 책임
  - 실행 가능성 관점에서 의견 제시

I (Input): 자문자
  - 전문 분야 의견 제공
  - 의사결정 권한 없음, 자문만

D (Decide): 결정자
  - 최종 결정 권한
  - 교착 시 결정 의무

예시 매핑:
  R: 시니어 개발자 (기술 분석)
  A: 보안팀 (보안 거부권)
  P: 개발팀 (구현 가능성)
  I: UX팀, QA팀 (전문 자문)
  D: CTO (최종 결정)
```

## Pattern 4: 조건부 합의 (Conditional Consensus)

완전한 합의가 어려울 때 조건부로 진행.

```
형식:
  "우리는 [결론]에 동의합니다.
   단, 다음 조건 하에서:
   1. [조건 1]이 충족되는 경우
   2. [조건 2]가 검증된 후
   3. [시점]에 재평가"

예시:
  "Next.js App Router 도입에 동의합니다.
   단, 다음 조건 하에서:
   1. 기존 Pages Router 점진적 마이그레이션 (6개월 내)
   2. 성능 벤치마크에서 기존 대비 10% 이상 개선 확인
   3. 3개월 후 팀 DX 만족도 재조사"

관리:
  - 조건별 책임자 지정
  - 검증 기한 설정
  - 조건 미충족 시 대안 행동 정의
```

## 교착 상태 해소

### 교착 상태 탐지

```
탐지 기준:
  1. 동일 논점이 2라운드 이상 반복
  2. 각 에이전트의 확신도 변화 < 5%
  3. 새로운 증거/논거 없이 기존 주장 반복
  4. 라운드 간 포지션 변화 없음

자동 탐지 로직:
  if (round > 2) {
    const positionChange = comparePositions(round[n], round[n-1]);
    const newEvidence = countNewEvidence(round[n]);

    if (positionChange < 0.05 && newEvidence === 0) {
      triggerDeadlockResolution();
    }
  }
```

### 해소 전략

#### 전략 1: 쟁점 분해

```
교착 주제: "서버리스 vs 컨테이너"

분해:
  하위 쟁점 1: 비용 (월 $500 이하 트래픽)
    → 합의: 서버리스가 저렴 ✓

  하위 쟁점 2: 성능 (P99 레이턴시)
    → 합의: 컨테이너가 안정적 ✓

  하위 쟁점 3: 개발 편의성
    → 미합의: 팀 경험에 따라 다름

  하위 쟁점 4: 확장성
    → 합의: 양쪽 모두 가능 ✓

결론: 3/4 합의 달성.
하위 쟁점 3은 "팀 숙련도 기반 선택" 조건부 합의.
```

#### 전략 2: 기준 재정립

```
기존 기준: "최고의 기술"
  → 교착 (주관적, 측정 불가)

재정립된 기준:
  1. 3개월 내 MVP 출시 가능한가? (Yes/No)
  2. 팀원 전원 1주 내 온보딩 가능한가? (Yes/No)
  3. 월 운영 비용 $200 이하인가? (Yes/No)

객관적 기준으로 전환 → 데이터 기반 판단 가능
```

#### 전략 3: 파일럿 제안

```
합의 불가 시 실험으로 전환:

"양측 주장 모두 타당하므로, 2주간 파일럿을 진행합니다.
  - 옵션 A: [팀 1]이 서버리스로 기능 X 구현
  - 옵션 B: [팀 2]가 컨테이너로 기능 X 구현
  - 비교 기준: 개발 시간, 성능, 비용
  - 2주 후 결과 기반 최종 결정"
```

#### 전략 4: 투표 + 소수의견 기록

```
최종 투표:
  옵션 A: 3표
  옵션 B: 2표
  → 옵션 A 채택

소수 의견 기록:
  "옵션 B를 지지한 이유:
   1. [논거 1]
   2. [논거 2]
   향후 재평가 트리거: [조건]
   이 기록은 6개월 후 재검토"
```

## 합의 품질 검증

### 합의 건강성 체크

```
체크리스트:
  □ 모든 참여자가 최소 1회 이상 발언했는가?
  □ 반론이 충분히 제기되었는가? (최소 3개)
  □ 증거/데이터가 인용되었는가?
  □ 소수 의견이 기록되었는가?
  □ 실행 가능한 구체적 결론인가?
  □ 리스크가 식별되었는가?
  □ 다음 단계가 명확한가?
```

### 합의 유형 분류

```
Strong Consensus (강한 합의):
  - 전원 동의
  - 높은 확신도 (평균 8+/10)
  - 충분한 증거 뒷받침
  → 즉시 실행 권장

Moderate Consensus (보통 합의):
  - 다수 동의 (2/3+)
  - 중간 확신도 (평균 6-8/10)
  - 일부 조건부
  → 조건 확인 후 실행

Weak Consensus (약한 합의):
  - 과반 동의 (1/2+)
  - 낮은 확신도 (평균 5-6/10)
  - 다수 조건부
  → 파일럿 또는 추가 조사

No Consensus (합의 실패):
  - 과반 미달
  - 근본적 가치관 차이
  → 의사결정자 판단 또는 분할 접근
```

## 안티패턴과 해결

| 안티패턴 | 설명 | 해결 |
|----------|------|------|
| Groupthink | 갈등 회피로 비판 없이 합의 | Devil's Advocate 강제 배치 |
| False Consensus | 침묵을 동의로 해석 | 모든 에이전트 명시적 입장 표명 의무 |
| Anchoring | 첫 의견에 과도하게 영향 | Round 1 독립 분석 후 동시 공개 |
| Sunk Cost | 이미 투자한 시간 때문에 고집 | 라운드별 리셋 가능 명시 |
| Scope Creep | 토론 범위 확장 | 모더레이터의 범위 관리 |
| Perfectionism | 100% 합의 추구로 교착 | 80% 합의면 진행, 나머지는 조건부 |
| Recency Bias | 마지막 발언에 과도한 비중 | 전체 토론 요약 후 투표 |
| Confirmation Bias | 자기 입장 뒷받침 증거만 인용 | 반대 증거 1개 이상 인용 의무 |

## 실전 시나리오

### 시나리오 1: 기술 스택 결정이 수렴하지 않을 때

```
상황: React vs Vue 논쟁이 3라운드째 교착

해소 프로세스:
  1. 쟁점 분해: 성능, DX, 생태계, 채용, 학습곡선
  2. 각 하위 쟁점별 독립 평가 (1-10점)
  3. 가중치 합의 (비즈니스 우선순위 기반)
  4. 가중 합계로 객관적 판단

결과:
  | 기준 | 가중치 | React | Vue |
  |------|:------:|:-----:|:---:|
  | 생태계 | 30% | 9 | 7 |
  | DX | 25% | 7 | 8 |
  | 채용 | 25% | 9 | 6 |
  | 성능 | 20% | 8 | 8 |
  | 합계 | 100% | 8.35 | 7.15 |

  → React 채택 (정량 근거), Vue 지지자 조건부 동의
  조건: "Vue Composition API와 유사한 패턴 채택"
```

### 시나리오 2: 보안 vs 사용성 갈등

```
상황: 보안팀은 MFA 강제, 제품팀은 간편 로그인 주장

해소 프로세스:
  1. 이분법 탈피: "보안 OR 사용성" → "보안 AND 사용성"
  2. 제3의 안 모색: 위험 기반 적응형 인증
  3. 조건부 합의:
     - 일반 로그인: 이메일 + 비밀번호 (사용성)
     - 민감한 작업: MFA 추가 (결제, 설정 변경)
     - 이상 징후: 자동 MFA 트리거 (보안)

결과: 양측 핵심 가치 보존, 사용자 경험 최적화
```

### 시나리오 3: 데드라인 임박한 교착

```
상황: 배포 2시간 전, 테스트 전략 미합의

해소 프로세스:
  1. 시간 제약 인정: "완벽한 합의 불가"
  2. 최소 합의: "이것만은 반드시" 목록 작성
     - 스모크 테스트 5개 (전원 합의)
     - 리그레션 테스트 10개 (전원 합의)
  3. 미합의 사항: 배포 후 1주 내 재논의
  4. 리스크 수용: "E2E 테스트 부족" 명시적 기록

결과: 제한된 시간 내 최선의 결정, 리스크 투명하게 관리
```
