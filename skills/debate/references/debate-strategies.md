# Debate Strategies

에이전트 토론의 전략, 역할 설계, 논거 구성, 수렴 기법에 대한 심화 가이드.

---

## 토론 프레임워크 선택

### 프레임워크 비교

| 프레임워크 | 구조 | 최적 상황 | 라운드 | 장점 |
|-----------|------|----------|:------:|------|
| 자유 토론 | 비구조적 | 브레인스토밍 | 2-3 | 창의적 아이디어 |
| 링컨-더글라스 | 1:1 대립 | 이분법적 결정 | 3-5 | 깊은 분석 |
| 의회식 | 찬반 팀 | 정책 결정 | 4-6 | 다각적 검토 |
| 라운드 로빈 | 순환 발언 | 이해관계자 분석 | 3-4 | 균등한 기여 |
| 수렴 토론 | 발산→수렴 | 최적안 도출 | 3 | 빠른 합의 |

### 상황별 추천

```
기술 스택 선정 → 링컨-더글라스
  이유: A vs B 명확한 대립 구도, 깊은 비교 가능

아키텍처 설계 → 의회식
  이유: 성능팀/보안팀/UX팀 등 다양한 관점 필요

UI/UX 방향 → 라운드 로빈
  이유: 디자이너, 개발자, 사용자 관점 균등 반영

비용 최적화 → 수렴 토론
  이유: 빠르게 실행 가능한 안 도출 목적

신규 기능 구상 → 자유 토론
  이유: 제약 없는 아이디어 발산
```

## 역할 설계 패턴

### Pattern 1: 전문가 패널

특정 도메인 전문가를 배치하여 전문성 기반 토론.

```
역할 구성 예시 - "모노레포 도입 여부"

Expert 1: 인프라 엔지니어
  관점: CI/CD 복잡성, 빌드 시간, 배포 파이프라인
  편향: 운영 비용 최소화

Expert 2: 프론트엔드 리드
  관점: 코드 공유, DX, 패키지 관리
  편향: 개발 생산성

Expert 3: 보안 담당자
  관점: 접근 제어, 의존성 관리, 감사 추적
  편향: 보안 정책 준수

Moderator: 기술 이사
  역할: 비즈니스 목표 기준 판단, 합의 도출
```

### Pattern 2: 드 보노 6색 모자

```
역할 매핑:

White Hat (사실): 데이터 분석가
  "벤치마크에 따르면 Turborepo는 빌드 시간 40% 단축..."

Red Hat (직감): UX 리서처
  "팀이 이 변화에 저항감을 느낄 수 있습니다..."

Black Hat (비판): 시니어 엔지니어
  "마이그레이션 중 2주간 생산성 저하가 예상됩니다..."

Yellow Hat (긍정): 제품 매니저
  "코드 공유로 기능 출시 속도 30% 향상 가능..."

Green Hat (창의): 스타트업 CTO
  "부분 도입으로 리스크를 줄이면 어떨까요?"

Blue Hat (진행): 모더레이터
  "지금까지 나온 논점을 정리하겠습니다..."
```

### Pattern 3: 악마의 변호사

모든 제안에 반론을 제기하는 전용 역할.

```
구성:
  - Proposer: 방안 제시
  - Devil's Advocate: 모든 약점, 실패 시나리오 지적
  - Mediator: 반론을 수용한 개선안 도출

예시 흐름:

Proposer: "Supabase로 인증을 구현합시다"
Devil's Advocate: "Supabase 장애 시 전체 서비스 중단. Lock-in 리스크.
  이메일 커스터마이징 제한. 소셜 로그인 공급자 제한적."
Mediator: "유효한 우려. 대안 검토: Auth 레이어 추상화로
  Lock-in 완화, Sentry로 장애 모니터링, 폴백 전략 수립."
```

### Pattern 4: 이해관계자 시뮬레이션

```
구성 - "가격 정책 변경"

역할 1: 기존 무료 사용자
  "무료 기능 축소는 배신감을 줍니다. 대안 서비스로 이동 고려."

역할 2: 유료 구독자
  "프리미엄 기능 차별화가 부족하면 구독 갱신 의문."

역할 3: 영업팀
  "엔터프라이즈 고객은 가격보다 기능과 SLA를 중시합니다."

역할 4: CFO
  "현재 번 레이트로는 18개월 내 수익화 필수."

합의 도출: 각 이해관계자의 최소 수용 조건 파악 → 교집합
```

## 논거 구성 기법

### 구조화된 주장

```
PREP 구조:
  P (Point): 핵심 주장을 한 문장으로
  R (Reason): 왜 그런지 이유 2-3개
  E (Evidence): 데이터, 사례, 벤치마크
  P (Point): 주장 재진술

예시:
  P: "React Server Components를 도입해야 합니다"
  R: 1. 초기 로딩 시간 단축 2. 번들 크기 감소 3. 서버 리소스 활용
  E: "Next.js 14 RSC 벤치마크에서 TTFB 60% 개선, JS 번들 40% 감소"
  P: "따라서 RSC 도입은 사용자 경험과 성능 모두에 긍정적입니다"
```

### 정량적 비교 매트릭스

```
| 기준 | 가중치 | 옵션 A | 옵션 B | 옵션 C |
|------|:------:|:------:|:------:|:------:|
| 성능 | 30% | 9 (2.7) | 7 (2.1) | 8 (2.4) |
| 비용 | 25% | 6 (1.5) | 9 (2.25)| 7 (1.75)|
| DX | 20% | 8 (1.6) | 6 (1.2) | 7 (1.4) |
| 생태계 | 15% | 9 (1.35)| 5 (0.75)| 7 (1.05)|
| 보안 | 10% | 7 (0.7) | 8 (0.8) | 8 (0.8) |
| 합계 | 100% | 7.85 | 7.10 | 7.40 |

결론: 옵션 A가 정량 분석 기준 최적
단, 비용 가중치 30%로 변경 시 옵션 B 역전 → 민감도 분석 필요
```

### 반론 대응 패턴

```
1. 수용 후 전환 (Acknowledge & Pivot)
  "맞습니다, 초기 비용이 높습니다. 그러나 6개월 ROI 관점에서..."

2. 조건부 수용 (Conditional Accept)
  "팀 규모가 5인 이하라면 동의합니다. 다만 10인 이상에서는..."

3. 재정의 (Reframe)
  "비용이 아닌 투자로 봐야 합니다. 기술부채 축적 비용 대비..."

4. 증거 제시 (Counter-evidence)
  "실제 사례를 보면, A사는 이 접근으로 장애율 80% 감소..."

5. 스케일 전환 (Scale Shift)
  "현재 규모에서는 문제없지만, 월 100만 MAU 도달 시..."
```

## 토론 라운드 설계

### 3라운드 기본 구조

```
Round 1: 입장 개진 (Divergence)
  - 각 참여자 독립적으로 입장 발표
  - 핵심 주장 + 근거 3개 이상
  - 다른 참여자 발언 참조 금지
  - 목적: 최대한 다양한 관점 수집

Round 2: 교차 검증 (Challenge)
  - 다른 참여자의 약점 지적
  - 자기 주장의 반론 방어
  - 새로운 증거/데이터 추가
  - 목적: 논리적 허점 발견

Round 3: 합의 도출 (Convergence)
  - 공통 합의 사항 도출
  - 미합의 사항에 대한 권고
  - 최종 추천안 + 리스크 정리
  - 목적: 실행 가능한 결론
```

### 5라운드 심화 구조

```
Round 1: 개별 분석 (각 에이전트 독립 작업)
Round 2: 입장 발표 + 질의
Round 3: 반론 및 방어
Round 4: 모더레이터의 쟁점 정리 + 재토론
Round 5: 최종 합의 + 실행 계획
```

### 수렴 기준 설정

```
자동 종료 조건:
  - 모든 참여자 동일 결론 도달
  - 3라운드 연속 새 논점 없음
  - 미리 설정한 합의율 달성 (예: 80%)

강제 종료 조건:
  - 최대 라운드 수 도달
  - 시간 제한 초과
  - 교착 상태 3회 반복
```

## 토론 품질 평가

### 평가 매트릭스

| 항목 | 가중치 | 기준 |
|------|:------:|------|
| 논거 다양성 | 25% | 서로 다른 관점의 수 |
| 증거 기반 | 25% | 데이터/사례 인용률 |
| 반론 품질 | 20% | 논리적 반박의 깊이 |
| 합의 실용성 | 20% | 실행 가능한 결론 도출 |
| 새로운 통찰 | 10% | 예상치 못한 발견 |

### 안티패턴 탐지

```
1. 에코 체임버 (Echo Chamber)
  증상: 모든 에이전트가 처음부터 동의
  해결: Devil's Advocate 역할 강제 배치

2. 권위 편향 (Authority Bias)
  증상: "시니어" 역할 의견에 무조건 동의
  해결: 익명 투표 라운드 추가

3. 정보 폭포 (Information Cascade)
  증상: 첫 발언자 의견에 연쇄 동의
  해결: Round 1에서 독립 분석 후 동시 공개

4. 분석 마비 (Analysis Paralysis)
  증상: 5라운드 이상 반복, 결론 없음
  해결: 강제 수렴 라운드 + 다수결

5. 허수아비 논증 (Straw Man)
  증상: 상대 주장을 왜곡하여 반박
  해결: 반박 전 상대 주장 요약 의무화
```

## 토픽별 토론 템플릿

### 기술 스택 선정

```yaml
topic: "프로젝트 X의 기술 스택 선정"
framework: lincoln-douglas
rounds: 4

roles:
  - name: "React/Next.js 옹호자"
    bias: "프론트엔드 생태계, DX"
    constraints: "반드시 서버 사이드 성능 데이터 포함"

  - name: "대안 프레임워크 옹호자"
    bias: "번들 크기, 런타임 성능"
    constraints: "생태계 크기와 채용 시장 고려 필수"

  - name: "풀스택 아키텍트 (모더레이터)"
    bias: "장기 유지보수성"
    constraints: "양측 의견 요약 후 판단"

evaluation_criteria:
  - 성능 벤치마크 인용 필수
  - 팀 규모(5인) 고려 필수
  - 6개월 내 MVP 출시 조건 반영
```

### 아키텍처 결정

```yaml
topic: "마이크로서비스 vs 모놀리스"
framework: parliament
rounds: 5

roles:
  - name: "마이크로서비스팀"
    members: ["백엔드 리드", "DevOps 엔지니어"]
    initial_position: "분리된 서비스가 확장성 보장"

  - name: "모놀리스팀"
    members: ["CTO", "주니어 개발자"]
    initial_position: "초기 단계에서 복잡성 최소화"

  - name: "의장"
    role: "쟁점 정리, 투표 진행"

constraints:
  - 현재 트래픽: 일 10,000 요청
  - 팀 규모: 4명
  - 예산: 월 $500 인프라
  - 목표: 1년 내 100배 성장 대응
```

### 비즈니스 결정

```yaml
topic: "프리미엄 가격 정책"
framework: round-robin
rounds: 3

roles:
  - name: "제품 매니저"
    focus: "사용자 가치, 전환율"
  - name: "CFO"
    focus: "수익, 마진, 번레이트"
  - name: "마케팅 리드"
    focus: "시장 포지셔닝, 경쟁사 가격"
  - name: "고객 대변인"
    focus: "가격 민감도, 이탈 리스크"

deliverable:
  - 가격 구간 3개 제안 (각 근거 포함)
  - 리스크 평가 (가격별 이탈률 예측)
  - A/B 테스트 계획
```

1. [Action Item 1] - 담당: [역할] - 기한: [날짜]

## 성능 최적화 팁

### 에이전트 수 가이드

| 토론 복잡도 | 에이전트 수 | 라운드 | 예상 시간 |
|------------|:----------:|:------:|:---------:|
| 단순 (A vs B) | 2-3 | 2-3 | 1-2분 |
| 중간 (다관점) | 3-4 | 3-4 | 3-5분 |
| 복잡 (이해관계자 多) | 4-6 | 4-6 | 5-10분 |

### 비용 절감

```
역할별 모델 선택:
  - 모더레이터: opus (판단력 필요)
  - 주요 토론자: sonnet (균형)
  - 보조 토론자: haiku (빠른 반응)

라운드별 최적화:
  - Round 1 (발산): sonnet (병렬 실행)
  - Round 2 (교차): sonnet (순차 실행)
  - Round 3 (수렴): opus (최종 판단)
```
