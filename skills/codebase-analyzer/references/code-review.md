# Code Review Guide

코드 품질 분석을 위한 상세 지침. 체계적인 리뷰와 피드백을 위한 체크리스트와 패턴을 제공합니다.

---

## 핵심 체크리스트

```markdown
[ ] 기능성: 코드가 의도대로 정확히 동작하는가?
[ ] 디자인: 시스템 구조와 잘 통합되며, SOLID 원칙을 따르는가?
[ ] 가독성: 코드가 명확하고 이해하기 쉬우며, 수정하기 용이한가?
[ ] 성능: 비효율적인 알고리즘, N+1 쿼리 문제 없이 효율적인가?
[ ] 보안: SQL 인젝션, XSS 등 취약점으로부터 안전한가?
[ ] 테스트: 변경 사항을 충분히 검증하는 테스트가 포함되어 있는가?
[ ] 문서화: 코드 변경에 따른 주석이나 문서가 업데이트되었는가?
```

---

## 분석 영역

### 1. 기능성 (Functionality)
- PR 요구사항 충족 여부
- 엣지 케이스 고려 (null, 빈 리스트, 최대/최소값)
- 동시성 문제 가능성
- 버그 수정 시 재현 테스트 추가 여부

### 2. 디자인 및 아키텍처
- SOLID 원칙 준수
  - SRP: 클래스/함수가 하나의 책임만 가지는가?
  - OCP: 확장에 열려있고 수정에 닫혀있는가?
  - LSP: 하위 타입이 상위 타입을 대체할 수 있는가?
  - ISP: 사용하지 않는 인터페이스에 의존하지 않는가?
  - DIP: 추상화에 의존하고 있는가?
- DRY 원칙: 중복 코드가 추출되었는가?
- 논리적 모듈 구성

### 3. 가독성 및 유지보수성
- 변수/함수/클래스 이름의 명확성
- 코딩 스타일 가이드 준수
- 코드 길이 (메서드 >200줄, 복잡한 블록 없는가)
- 매직 넘버 대신 상수 사용

### 4. 성능 최적화
- 비효율적 알고리즘 (O(n²) 루프)
- N+1 쿼리 문제
- 불필요한 JOIN, SELECT *
- 메모리 비효율 (큰 파일 한번에 로드)
- 캐싱 기회

### 5. 보안 취약점
- 입력 유효성 검사
- SQL 인젝션, XSS 방어
- 하드코딩된 시크릿/크레덴셜
- 민감 정보 로그 노출
- 인증/인가 로직

## 피드백 프레임워크

### Conventional Comments

| 라벨 | 의미 |
|------|------|
| `praise:` | 칭찬 및 긍정적 강화 |
| `nitpick:` | 사소한 수정 (Non-blocking) |
| `suggestion:` | 개선 제안 |
| `issue:` | 논리적 결함 (Blocking) |
| `question:` | 의도 확인 및 질문 |
| `thought:` | 아이디어 및 기술적 논의 |

### Pn 룰 (우선순위)

| 태그 | 의미 | 처리 |
|------|------|------|
| P1 | 필수 반영 (Critical) | 반드시 수정 |
| P2 | 적극 권장 | 미반영 시 토론 필요 |
| P3 | 수용 권장 | 어려우면 후속 작업 생성 |
| P4 | 참고 의견 | 작성자 판단에 맡김 |
| P5 | 사소한 의견 | 무시해도 괜찮음 |

## 출력 형식

```markdown
# Code Review Report

## Overall Score: [X/10]

## Strengths
- [긍정적인 측면 나열]

## Issues Found

### Critical (P1)
- [반드시 수정해야 할 이슈]

### Major (P2)
- [중요하지만 긴급하지 않은 이슈]

### Minor (P3-P5)
- [개선하면 좋은 사항]

## Security Concerns
- [보안 취약점 목록]

## Performance Suggestions
- [성능 최적화 권장사항]

## Priority Actions
1. [가장 중요한 액션]
2. [두 번째 우선순위]
3. [세 번째 우선순위]
```

---

## 실행 지침

1. 분석할 파일 또는 디렉토리 확인
2. 핵심 체크리스트 기반으로 코드베이스 스캔
3. Pn 룰로 이슈 우선순위 지정
4. 개선 제안에 코드 예시 포함
5. 심각도별 발견사항 정리하여 리포트 생성
