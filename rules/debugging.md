# Debugging

## 원칙

에러가 발생하면 **찍어맞추기(guess-and-check)를 금지**한다. 반드시 4단계 프로세스를 따른다.

## 4단계 디버깅 프로세스

### 1. 관찰 (Observe)
- 에러 메시지 **전문**을 읽는다 (첫 줄만 읽지 않는다)
- 스택 트레이스를 분석하여 실제 발생 위치를 파악한다
- 재현 조건을 명확히 한다 (언제, 어떤 입력에서 발생하는가)

### 2. 가설 (Hypothesize)
- 가능한 원인 후보를 **최소 2개** 나열한다
- 각 후보에 대해 **왜 그렇게 생각하는지 근거**를 명시한다
- 가장 유력한 후보부터 검증한다

### 3. 검증 (Verify)
- 가설별 검증 방법을 설계한다
- 로그, 브레이크포인트, 단위 테스트, `console.log` 등으로 증거를 수집한다
- **증거가 확인된 원인만** 수정 대상으로 판단한다

### 4. 수정 (Fix)
- 한 번에 **하나의 변경만** 적용한다
- 수정 후 원래 에러를 **재현 시도하여 해결을 확인**한다
- 수정이 새로운 문제를 만들지 않았는지 관련 테스트를 실행한다

## 금지 행위

- ❌ 에러 메시지를 읽지 않고 코드 수정 시작
- ❌ "일단 이것저것 바꿔보기" (산탄총 디버깅)
- ❌ 한 번에 여러 곳 동시 수정 (어떤 변경이 효과가 있었는지 모름)
- ❌ 수정 후 재현 테스트 생략 ("아마 고쳐졌을 거다")
- ❌ 에러를 숨기는 수정 (try-catch로 삼키기, 에러 무시)
